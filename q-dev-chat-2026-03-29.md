## 比較

**収入・支出それぞれに更新ボタン（分ける）**
- メリット: 片方だけ変更した場合に、もう片方のAPI呼び出しが不要
- デメリット: ボタンが2つになり、画面がやや煩雑

**1つの更新ボタン（まとめる）**
- メリット: 操作がシンプル
- デメリット: 片方しか変更していなくても両方のAPIを呼び出す（ただし、カテゴリ数は少ないため実質的なコスト差はほぼなし）

---

## 推奨

**1つの更新ボタン**を推奨します。

カテゴリ数は多くても数十件程度であり、API呼び出しコストの差は無視できます。操作のシンプルさを優先する方が使い勝手が良いと考えます。

## 共通化の検討

### 現状の構成
Lambda関数はSAMの構成上、**関数ごとに独立したディレクトリ**になっています。

```
lambda/src/
  ├── transaction/
  │   └── index.js
  ├── bank-account/
  │   └── index.js
  └── ...
```

### 共通化の方法

**共通モジュールファイルを作成する**

```
lambda/src/
  ├── shared/
  │   └── dailyBalanceHelper.js  ← 共通処理
  ├── transaction/
  │   └── index.js
  ├── bank-account/
  │   └── index.js
  └── ...
```

`dailyBalanceHelper.js` に日次残高更新処理をまとめ、各 `index.js` から `require('../shared/dailyBalanceHelper')` で呼び出す。

### 注意点
- SAMのデプロイ時、各関数の `CodeUri` が個別ディレクトリ（例: `src/transaction/`）を指しているため、**`shared` フォルダが各関数のパッケージに含まれない**可能性がある
- `template.yaml` の `CodeUri` を各関数から `src/` 全体に変更する必要がある

## `transaction/index.js` と `daily-balance/index.js` の比較

### 日次残高更新の目的

| | `transaction/index.js` | `daily-balance/index.js` |
|---|---|---|
| 用途 | 取引1件の追加・更新・削除時のリアルタイム更新 | 指定期間の全取引を一括再計算 |
| 起動タイミング | 取引操作のたびに自動実行 | 手動で再計算ボタン押下時 |

---

### ロジックの違い

**前日残高の取得**
- `transaction/index.js` : 前日以前の最新残高を取得、なければ初期残高
- `daily-balance/index.js` : 開始日の前日以前の最新残高を取得、なければ初期残高
- → **ほぼ同じ**

**残高計算方法**
- `transaction/index.js` : 当日の全取引を再取得して合計計算 → 翌日以降を**差分**で更新
- `daily-balance/index.js` : 期間内の日付を順番に処理し、日ごとに累積計算 → 翌日以降も同様に累積
- → **異なる**（差分更新 vs 累積計算）

**対象範囲**
- `transaction/index.js` : 当日 + 翌日以降（無制限）
- `daily-balance/index.js` : 指定した開始日〜終了日のみ
- → **異なる**

**クレジットカード引落の扱い**
- `transaction/index.js` : クレジットカード取引は引落口座の銀行残高を直接更新
- `daily-balance/index.js` : クレジットカード取引を引落日に銀行口座の支出として累積計算
- → **アプローチが異なるが結果は同じ**

`transaction/index.js` のリアルタイム更新ロジックと `daily-balance/index.js` の再計算ロジックは**目的・方式ともに異なる**ため、共通化の対象は `transaction/index.js` と `bank-account/index.js`（今回追加分）の2つのみです。

## Gitリポジトリとの連携について

### Gitとは
ソースコードの**変更履歴を管理するツール**です。ローカルのファイルをGitで管理し、**GitHub**（クラウド上のリポジトリサービス）に保存することで、AWS Amplifyがそのコードを取得してデプロイできるようになります。

---

### 全体の流れ

```
ローカルPC                GitHub              AWS Amplify
(ソースコード) → push → (リポジトリ) → 連携 → (自動ビルド・公開)
```

---

### 具体的な手順

**1. GitHubアカウントを作成**（未作成の場合）
- [https://github.com/](https://github.com/) でアカウント作成

**2. ローカルにGitをインストール**（未インストールの場合）
- [https://git-scm.com/](https://git-scm.com/) からインストール

**3. GitHubに新しいリポジトリを作成**
- GitHub上で「New repository」をクリック
- リポジトリ名を入力（例: `household-budget-app`）
- Private（非公開）を推奨

**4. ローカルのコードをGitHubにpush**
```cmd
cd C:\household-budget-app
git init
git add .
git commit -m "initial commit"
git remote add origin https://github.com/ユーザー名/household-budget-app.git
git push -u origin main
```

**5. AWS AmplifyとGitHubを連携**
- Amplifyコンソールでリポジトリを選択するだけで自動デプロイが設定される

---

### メリット
- コードを push するたびに**自動でデプロイ**される
- 変更履歴が残るので**以前の状態に戻せる**
- `.env` ファイルは `.gitignore` に含まれているため、**APIキー等が公開されない**

## Amplify Hostingのビルドの仕組み

### 基本的な動作

```
GitHubにpush → Amplifyが変更を検知 → 自動でビルド → 自動でデプロイ
```

**GitHubに push するたびに**Amplifyがビルドを実行します。

---

### ビルドが走るタイミング

| 操作 | ビルド実行 |
|---|---|
| GitHubにpush | ✅ 自動実行 |
| 手動でビルド実行 | ✅ コンソールから可能 |
| ファイルを変更しただけ（pushなし） | ❌ 実行されない |

---

### 毎回ビルドされることへの影響

**費用面**
- 1回のビルドは通常2〜3分程度
- 月1,000分の無料枠があるので、1日1回pushしても月30分程度で全く問題なし

**時間面**
- ビルド完了まで約2〜5分かかる
- ビルド中も**現在公開中のサイトはそのまま動き続ける**
- ビルド完了後に新しいバージョンに切り替わる

## パスワード保護の設定手順

### 手順

1. [https://console.aws.amazon.com/amplify/](https://console.aws.amazon.com/amplify/) を開く

2. `household-budget-app` をクリック

3. 左メニューの「ホスティング」→「アクセスコントロール」をクリック

4. 「アクセス設定を管理」ボタンをクリック

5. `master` ブランチの行で「アクセスレベル」を **「制限あり」** に変更

6. 以下を入力して保存
   - ユーザー名: 任意（例: `admin`）
   - パスワード: 任意（推測されにくいものを推奨）

7. 「保存」をクリック

---

## 理由

**HTTPS通信**では、ブラウザとサーバー間の通信が**TLS（SSL）で暗号化**されています。

```
ブラウザ → 暗号化 → インターネット → 復号 → Amplifyサーバー
```

Basic認証のユーザー名・パスワードも暗号化された通信の中に含まれるため、**盗聴されても解読できません。**

---

## 注意点

ただし以下の点は意識しておいてください。

| 項目 | 内容 |
|---|---|
| HTTPS | ✅ 通信は暗号化されている |
| パスワードの強度 | 推測されにくいパスワードを設定する |
| パスワードの使い回し | 他のサービスと同じパスワードは避ける |
| URLの共有 | URLを不特定多数に公開しない |

---

個人利用の家計簿アプリであれば、**AmplifyのBasic認証 + HTTPS で十分なセキュリティ**と言えます。

## 開発フロー

```
コード修正
  ↓
npm run dev（ローカルで動作確認）
  ↓
問題なければ git push
  ↓
Amplifyが自動でビルド・デプロイ（2〜5分）
  ↓
本番URLで最終確認
```

## 補足

- バックエンド（Lambda）を変更した場合は別途 `sam build` → `sam deploy` も必要です
- フロントエンドのみの変更であれば `git push` だけでOKです
